# 1. 데이터베이스



### 1. 용어

* 개념적 데이터 모델 구성

  ​	<img src="https://user-images.githubusercontent.com/70613905/164577669-8921844c-d0c2-4a67-be68-bb1ae34f3e78.JPG" alt="image-20220422101719167" style="zoom:67%;" />

  * 개체(Entity)

    * DB에 표현하려는 대상
    * 교수, 학생, 자동차 등

  * 속성(Attribute)

    * 파일 구조 상 데이터 항목, 필드

    * 식별자(identifier)

      →   각 인스턴스를 유일하게 구분하는 구분자

      →   논리 데이터 모델링의 키

      →   주식별자(=> 기본 키), 보조 식별자(=> 유니크 키)

  * 관계(Relationship)

    * 개체 간 관계 혹은, 속성 간 논리적 연결

* 논리적 데이터 모델(관계형 데이터 모델)

​		<img src="https://user-images.githubusercontent.com/70613905/164576177-e6a297ee-5752-423b-9c58-9f7fe6458c2a.JPG" alt="image-20220422095056127" style="width:80%;" />

* 릴레이션(Relation)
  * 데이터를 표의 형태로 구성
  * 파일구조의 테이블
  * 릴레이션 스키마 + 릴레이션 인스턴스 로 구성
* 릴레이션 스키마
  * 릴레이션 이름, 각 애트리뷰트의 집합
  * 릴레이션에 데이터 넣기 위한 틀
* 릴레이션 인스턴스
  * 릴레이션 튜플의 집합
* 도메인
  * 한 애트리뷰터가 가질 수 있는 값의 범위
  * ex. 학년 애트리뷰트 도메인 = 1, 2, 3, 4
* 튜플(Tuple)
  * 릴레이션의 행
  * 파일구조의 레코드(Record)
* 속성(Attribute)
  * 릴레이션의 열



### 2. 데이터베이스

* 통합하여 관리되는 데이터 집합체

* 사용 이유

  * DB 이전에는 파일시스템으로 데이터 관리
  * 데이터가 커짐에 따라 파일로 관리하는 것에 한계
    * 파일 : 텍스트의 모음
    * DB : 구조화되어 항목 별로 값이 있음 => 방대한 데이터 체계적 관리

* 특징

  * 데이터 독립성

    * 물리적 독립성

      →   내부 스키마 변해도 개념 스키마 변하지 않음

      →   응용 프로그램 영향 없이 물리적 구조 변경 가능

      →   DB 사이즈 늘리기, 데이터 파일 추가 등

    * 논리적 독립성

      →   개념스키마가 변해도 외부 스키마 변하지 않음

      →   논리적 구조로 다양한 응용프로그램의 논리 구조 만족 가능

    * 스키마

      →   DB에 저장되는 데이터 구조와 제약조건 정의한 것

      ​	<img src="https://user-images.githubusercontent.com/70613905/164576184-9f60c21a-7ef1-4b77-a4ed-5292a7ffc0d8.JPG" alt="image-20220422091613749" style="width:80%;" />

      →   개념 스키마 / 스키마

      >* DBA/조직 전체 관점의 스키마
      >* DB 전체 골격 구조
      >* DB마다 1개
      >* 개체/관계/제약조건 등
      >* ex. Student라는 개체 정의

      →   외부 스키마 / 서브 스키마

      >* 외부(사용자)가 보는 스키마
      >
      >  →   프로그래머는 외부 스키마 구조를 보고 DML을 이용해 데이터 조작
      >
      >* 사용자가 보는 관점에 따라 여러 스키마 존재
      >
      >* ex. 네이버 DB - 네이버 메일 스키마, 네이버 카페 스키마, 네이버 블로그 스키마...
      >
      >* ex. Student 개체를 이용하여, 두 사용자(학적과, 학생과)가 자신의 데이터 구조 정의

      →   내부 스키마

      >* 저장장치 관점 스키마
      >
      >  →   물리 장치에 데이터가 실제로 저장되는 방법 표현
      >
      >* 저장될 내부 레코드 형식, 데이터 항목 표현 방법 등
      >
      >* ex. Student 개체의 레코드 형식과 인덱스 정의

  * 데이터 무결성

    * DB는 제약조건 등을 통해 데이터 무결성 유지

  * 데이터 보안성

    * 계정, 권한 관리를 통해 데이터 보안 유지

  * 데이터 일관성

    * 제약조건 등을 통해 데이터 일관되게 유지

  * 데이터 중복 최소화

    * 데이터를 통합하여 관리함으로써 데이터 중복 최소화

* 데이터베이스 성능

  * 디스크 IO를 얼마나 줄이냐가 관건

    * 디스크 IO

      →   디스크 드라이브의 플래터(원판)을 돌려 목적 데이터 위치로 디스크 헤더 이동시킨 후, 데이터 읽는 것

      →   이 때, 데이터 읽는 소모시간은 디스크 헤더의 이동시간에 결정됨

      →   즉, 디스크 성능 = 디스크 헤더 위치 이동 없이 얼마나 많은 데이터 한 번에 읽느냐

      →   때문에 DB 쿼리 튜닝은 랜덤IO를 줄이는 것이 목적 (순차IO로)
